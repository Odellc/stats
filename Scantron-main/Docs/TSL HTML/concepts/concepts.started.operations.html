Getting Started - Reader Operations

Overview

Reader Operations have been introduced as a high-level API for working with UHF RFID Readers. A IReader has a number of IReaderOperation.

These operations define a high level reader task which include:

  • Performing an RFID inventory when the trigger of the reader is pressed
  • Performing a barcode scan when the trigger of the reader is pressed
  • Commanding the barcode engine on the host terminal when the trigger of the reader is pressed
  • Performing an RFID inventory on command
  • Performing a Tag Access operation (read / write / lock / kill / blockpermalock etc...) on command
  • Reporting periodic reader status

These operations are more declarative than using the Technology Solutions UK Ltd ASCII 2 Protocol directly and their use is covered in the TechnologySolutions.Rfid namespace summary and in the Reader Operations sample code.

One concern left to the developer is that the reader is only capable of running one instance of the same command at any one time. For this reason you should only have one operation enabled using a command at any one time.

If you have an inventory running on trigger press and then perform a commanded inventory the trigger press inventory will be impacted.

If you have a inventory running on trigger press and then perform a commanded read the two commands will not affect each other.

Reader Operations for ASCII Protocol Readers

A reader operation can be started in a few ways. It can also just run once or be repeated at a specified interval.

A hardware trigger is where the operation is run on the reader and messages are sent back to the host to report progress. An ASCII Protocol reader can only have one primary and one secondary trigger operation. The last enabled operation for each trigger is the one assigned to the trigger. Enabling a new operation assigned to the trigger will replace the previous although the previous operation should be disabled first.

A software trigger is where the operation is commanded from the host in response to the reader reporting changes in trigger state. As these operations are commanded from the host you could have multiple operations for the same trigger but this is not advised as it will be confusing for an operator.

A manual trigger is where the operation is started from code. The StartAsync() will not return until the task is completed (or canceled with StopAsync() ). If starting a periodic task ensure to use @System.Threading.Tasks.Task.Run or similar. When running a single reader operation (e.g. to write a transponder) you can simple await StartAsync() and follow with an await StopAsync()

Barcode

IReaderOperationBarcode performs a barcode read per trigger press. The trigger is assigned using TriggerIndex. This command is hardware triggered so cannot be repeated.

For ASCII Protocol readers the trigger is either the primary /single click (TriggerIndex = 1) or secondary double click (TriggerIndex = 2).

to implement this command .sa Switch Action Command is used to set the trigger action to a .bc Barcode Command and a barcode responder is added to the responder chain

Battery Status

IReaderOperationBatteryStatus runs periodically to report the reader battery status. This runs periodically at the specified TriggerInterval. It is a manual operation but has a TriggerInterval so can be left running while the reader remain connected.

For ASCII Protocol readers this command uses the .bl Battery Level Command command to obtain the battery and charge status.

Find Tag

IReaderOperationFindTag is used to locate a transponder. This command is hardware triggered.

For ASCII Protocol readers that support the .ft FindTag Command this will be used. Where the .ft FindTag Command is not supported the operation falls back to the .iv Inventory Command.

The state of the operation is reported with TagFindState and TagFindStateChanged.

The proximity to the transponder(s) identified by the Filter is reported by the @TechnologySolutions.Rfid.IReaderOperationFingTag.TagFindUpdate. This reports the highest discovered signal strength to a matching transponder and whether more than one transponder is responding.

Host Barcode

IReaderOperationHostBarcode is used to trigger a barcode engine on the host terminal from the trigger on the reader via IHostBarcodeHandler . This operation will cancel any hardware operation assigned to its TriggerIndex but is run as a software trigger.

As the trigger is pressed it will call InitiateScan() and as the trigger is released it will call CancelScan().

Inventory

IReaderOperationInventory is used to report transponders on trigger press. This operation is hardware triggered.

For ASCII Protocol reader this is installed as a .iv Inventory Command on the switch using .sa Switch Action Command

Transponders

IReaderOperationTransponders is used for perform inventory operations as a manual OR a software trigger. Specify the Filter to identify the transponders to return and the Report to identify what fields to return per transponder.

Transponder Access

IReaderOperationTranspondersAccess is used for perform Tag Access operations as a manual OR a software trigger. Specify the Filter to identify the transponders to return and the Report to identify what fields to return per transponder. Specify the Access to identify the details of the Tag Access to perform.
